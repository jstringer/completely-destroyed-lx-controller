# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`lxcontrol` is a MIDI-to-Art-Net lighting controller built on the [NAP Framework](https://nap-framework.tech) ("Not A Program" — a real-time C++ platform for graphics/interactive applications), located at `apps/lxcontrol` inside a full NAP SDK installation (`NAP-0.8.0-Win64-x86_64`). NAP apps are declarative-first: behavior is composed from JSON resource/entity/component graphs (`data/objects.json`) and driven by native C++ code only where JSON composition isn't enough.

It receives MIDI input and translates it into Art-Net/DMX output to drive three identical Eurolite LED Strobe SMD PRO 540 fixtures (run in 22-channel DMX mode), with GUI-editable fixture values, and a live in-app authoring UI for Presets, Effects, and MIDI mappings — created, edited, and deleted at runtime, not hand-authored in JSON.

## Repository layout

- `src/` — the application itself (`lxcontrolApp`), owns services/scene/window and drives update/render/input, and draws the tabbed authoring GUI (Fixtures / Presets / Effects / MIDI).
- `module/` — the `naplxcontrol` module containing all lxcontrol-specific resource/component classes (fixtures, effect layers, presets, MIDI mappings) plus `lxcontrolService`, the runtime authority described below. App code and module code are separate compilation units with separate `CMakeLists.txt`/`*.json` manifests — don't mix them up.
- `data/objects.json` — the resource/entity/component graph for the **fixed rig only**: fixtures, the Art-Net controller, the MIDI input port, the wildcard MIDI listener entity, and the per-fixture value-editing GUIs. Loaded at runtime by the `ResourceManager`. Also editable visually in **Napkin** (NAP's visual JSON editor) — see the build gotcha below if you use it alongside this repo.
- `data/user_content.json` — **not hand-authored, not checked into a template.** Everything created live through the Presets/Effects/MIDI tabs (Presets, EffectLayers + their SequencePlayers/overrides, MidiMappings) is serialized here by `lxcontrolService::save()` and reloaded here at startup. `objects.json` is never rewritten by the app. Delete this file to reset all authored content back to nothing.
- `data/shaders/` — GLSL shaders referenced by material resources in `objects.json`.
- `app.json` / `module/module.json` — declare required NAP modules and metadata. **Do not hand-edit `CMakeLists.txt`, `cached_app_json.cmake`, or `module/cached_module_json.cmake`** — they are generated from `app.json`/`module.json`; put custom CMake logic in `app_extra.cmake` / `module_extra.cmake` instead (create if absent).
- `msvc64/` — generated Visual Studio/CMake build tree. Never hand-edit; regenerate instead.

## Build / regenerate / package

All three operate on this app directory via NAP's shared Python buildsystem tooling (`../../tools/buildsystem/common/`) and the bundled Python interpreter (`../../thirdparty/python`):

```bat
regenerate.bat   REM regenerates CMake project files (msvc64/) from app.json/module.json — run after changing RequiredModules or manifests
build.bat        REM builds the app (invokes CMake/MSBuild under the hood)
package.bat      REM packages the app for distribution
```

To build directly with an already-generated solution, open `msvc64/lxcontrol.sln` in Visual Studio, or build via CMake against the `msvc64/` directory.

There is no separate lint/test command for this app — correctness is verified by building and running.

**Gotchas learned the hard way:**
- **Always run `regenerate.bat` after adding a new `.cpp` file to `module/src/`**, not just after changing `RequiredModules`. The Visual Studio project's file list is captured by CMake at `regenerate.bat` time; a new source file left out of that list means `build.bat` will report success (nothing changed, so nothing to build) while silently never compiling or linking it in. Symptom: `objects.json` fails to load with `Unknown object type <YourNewComponent> encountered`, even though the class is written correctly.
- **If Napkin (NAP's visual editor) has this project open, `build.bat` fails with `LNK1104: cannot open file ...naplxcontrol.dll`.** Napkin loads the module DLL to introspect its RTTI types and holds a lock on it. Close the project in Napkin before rebuilding.
- In NAP's JSON schema, **`"Type"` is a reserved key** (used to name the resource's C++ class) and can't also be used as a custom property name — e.g. `FixtureChannel`'s DMX-value-kind property is exposed as `"ChannelType"`, not `"Type"`, for exactly this reason.
- **`Embedded` vs plain-reference properties matter and aren't visually distinguishable in C++.** A `ResourcePtr<T>`/`vector<ResourcePtr<T>>` property registered with `EPropertyMetaData::Embedded` (e.g. `ParameterGroup`'s `Parameters`/`Groups`, `SequencePlayer`'s `Clock`/`Outputs`) must be written as a full inline object definition in JSON; the same-looking property registered `Default` (nearly everything else, including all of this module's own properties) must be written as an `mID` string reference to a separately-declared object. Mixing them up produces `Encountered embedded pointer that points to a non-embedded object` with no indication of which property is at fault. **Concretely bit us once**: we tried to give a "root" `ParameterGroup` a `Groups` list of ID-string references to the three independently-declared per-fixture groups (`Strobe1_Params`/`Strobe2_Params`/`Strobe3_Params`), to get one `ParameterGUI`/one preset-snapshot call spanning all fixtures. `ParameterGroup.Groups` is Embedded, so this doesn't work — a group referenced there must be declared *only* inline, not also as its own top-level object. There is no merged "all fixtures" group in this codebase because of this; see Presets below for how cross-fixture presets actually work instead.
- **Runtime-constructed resources (via `ResourceManager::createObject<T>()`) need every init-order step called by hand.** Unlike JSON-loaded objects (where the `ResourceManager` calls `init()` — and, for `Device`s, `start()` — on everything in dependency order automatically), an object built in C++ at runtime only gets what you explicitly call. `lxcontrolService::createEffectLayer()` is the canonical example: it builds override `ParameterFloat`s → `SequencePlayerCurveOutput`s → `SequencePlayerStandardClock` → `SequencePlayer` (`init()` then `start()`, since it's a `Device`) → `EffectLayer` → `EffectLayerOverride`s, calling `init(errorState)` on each in that leaves-first order before wiring the next one.
- **Non-embedded (`Default`) pointer targets must be included explicitly when serializing a runtime object graph back to JSON.** `nap::rtti::serializeObjects(rootObjects, writer, errorState)` only writes what's in `rootObjects` (plus anything embedded inside them) — it does not recursively pull in `Default`-pointed-to objects. `lxcontrolService::save()` has to explicitly list every `Preset`, `EffectLayer`, `SequencePlayer`, `SequenceEditor`, `SequenceEditorGUI`, override `ParameterFloat`, `EffectLayerOverride`, and `MidiMapping` it owns — leaving one out produces a dangling ID reference in `user_content.json` that fails to resolve on the next load.
- **Never blanket-query `ResourceManager::getObjects<T>()` for a type that both `objects.json` and `user_content.json` might declare** (this bit us with `ParameterFloat` — the 66 static per-fixture base parameters live in `objects.json`; only the handful of runtime-created override parameters belong in `user_content.json`). `lxcontrolService` tracks exactly which objects it created/owns in its own `EffectLayerEntry` bookkeeping rather than re-deriving that set from a broad `getObjects<T>()` scan, specifically to avoid re-declaring a static object under a colliding `mID` and failing to load.
- Some SDK helper classes are not exported for out-of-module use even though they look like public API — e.g. `nap::MidiPortInfo` (`napmidi`) has no `NAPAPI` on its class declaration, so linking against it from app/module code fails with unresolved externals. The MIDI tab's device list uses the already-`NAPAPI`-exported `nap::MidiInputPort::getPortNames()` instead (the ports our statically-declared `MidiPort` resource opened at startup) — this only reflects startup state, since `napmidi` has no live/hot-plug enumeration API at all.
- **`IMGuiService::selectWindow(window)` must be called before `ParameterGUI::show()` / `SequenceEditorGUI::show()`**, and both must happen in `update()`, not `render()` (`render()` only draws what was queued during `update()`). Skipping `selectWindow()` crashes inside `napimgui.dll` rather than failing gracefully. See `lxcontrolApp::update()` and NAP's own `demos/sequencer/src/sequencerapp.cpp` for the reference pattern.

## Architecture

### Fixtures

A **FixtureType** (`module/src/fixturetype.h`) is a named channel layout, made of **FixtureChannel**s (`fixturechannel.h`: name + DMX offset + `ChannelType`). `data/objects.json` declares one shared `FixtureType_EuroliteStrobe540_22ch` (22 channels: Dimmer, Strobe, six RGB units, PresetColor, AutoSound — see the Eurolite manual for the channel table) and patches it into **three independent fixtures** — `Fixture_Strobe1`/`Fixture_Strobe2`/`Fixture_Strobe3` — on `Universe0` at sequential `StartChannel`s (0/22/44). Each fixture has its own `FixtureType` patch (`fixture.h`), its own **FixtureChannelBinding**s (`fixturechannelbinding.h`), and its own **ParameterGroup** (`Strobe1_Params`/`Strobe2_Params`/`Strobe3_Params`) of 22 base `ParameterFloat`s — there is deliberately no shared/merged control surface; the three lights are controlled independently even though Presets apply to all three at once (see below).

Each `FixtureChannelBinding` holds a *base* `nap::ParameterFloat` (0.0–1.0, GUI-editable and MIDI-mappable) plus an ordered list of **EffectLayerOverride**s (`effectlayeroverride.h`) — pairings of an `EffectLayer` with a dedicated override parameter. `FixtureChannelBinding::resolveValue()` returns the highest-*priority active* layer's override value, falling back to the base parameter. Overrides can be attached two ways: statically in JSON (`Overrides` property, resolved by `init()`), or at runtime via `FixtureChannelBinding::addOverride()` — the mechanism `lxcontrolService` uses when a live-authored Effect targets this channel; see Effects below. A `FixtureDmxWriterComponent` (`fixturedmxwritercomponent.h`) reads each `Fixture`'s resolved channel values every frame and calls `ArtNetController::send()`; `FixtureDriverEntity` in `objects.json` holds one writer component per fixture.

Each fixture also gets its own `nap::ParameterGUI` (`FixtureParameterGUI`/`FixtureParameterGUI2`/`FixtureParameterGUI3`, `Serializable: false` — preset save/load is handled by `lxcontrolService`, not `ParameterGUI`'s own popup) shown side-by-side in the app's **Fixtures** tab.

### Effects

An **EffectLayer** (`effectlayer.h`) wraps a `nap::SequencePlayer` (from NAP's `napsequence` module) plus an integer `Priority`. Multiple layers can run concurrently; when two active layers both override the same fixture channel, `FixtureChannelBinding` picks the higher-priority one (simple "first active layer wins" mixing — not true HTP/additive blending).

**Effects are created live**, in the app's **Effects** tab: pick a name, a priority, and check off which (fixture, channel) pairs it should drive — this is where the effect's targets are fixed; a Preset later just chooses whether this effect plays, it can't re-target it. Creating one calls `lxcontrolService::createEffectLayer()`, which builds (via `ResourceManager::createObject<T>()`, entirely in C++, no JSON) a `SequencePlayerStandardClock`, a `SequencePlayer` (`CreateEmptySequenceOnLoadFail: true`), one `SequencePlayerCurveOutput` + override `ParameterFloat` + `EffectLayerOverride` per chosen target, the `EffectLayer` itself, and a dedicated `SequenceEditor` + `SequenceEditorGUI` pair. The Effects tab's "Timeline" checkbox toggles that `SequenceEditorGUI`'s visibility (kept open across tab switches, shown from `lxcontrolApp::update()`) — actually authoring keyframes uses NAP's existing `nap::SequenceEditorGUI` mechanism unchanged, just per-effect and GUI-toggled instead of one always-on window per hand-declared layer. All of this is persisted to `data/user_content.json` and reconstructed (including replaying `addOverride()`, since that wiring itself isn't serializable) by `lxcontrolService::rebuildFromLoadedContent()` at startup.

### Presets

A **Preset** (`preset.h`) names a set of per-fixture value snapshots plus which `EffectLayer`s should be playing while it's active (`LayersToPlay`, checked via `Preset::playsLayer()`). Because there's no merged all-fixtures `ParameterGroup` (see the Embedded-property gotcha above), a Preset's snapshot is actually **one saved-parameter file per fixture** — `Preset::mPresetFile` holds a base id, and `lxcontrolService` derives each fixture's actual filename as `<base>_<FixtureID>.json` via `nap::ParameterService::savePreset()`/`loadPreset()` (the same SDK mechanism `ParameterGUI`'s save/load-preset popup uses, just driven directly instead of through that popup). A Preset always applies to *all* fixtures at once — there's no per-preset fixture subset — but each fixture's values within that preset are independent.

`lxcontrolService` (module/src/lxcontrolservice.h/.cpp) is the runtime authority for Presets — it replaces the SDK's `PresetManagerComponent`/`ParameterBlendComponent` pair used in earlier iterations of this app, because both have their preset/layer lists fixed at JSON-load time, which doesn't support presets created live in the GUI. `createPreset()`/`applyPreset()`/`resnapshotPreset()`/`removePreset()` on `lxcontrolService` do what those components used to: `applyPreset()` loads each fixture's snapshot instantly (no time-based blend — a deliberate simplification, worth revisiting if a crossfade is wanted later) and starts/stops every known `EffectLayer` per `preset->playsLayer()`. All of this happens from the app's **Presets** tab: "+ New Preset" snapshots current live values across all three fixtures; "Update Snapshot" re-captures them; the "Effects to play" tree lets you check which effects this preset activates; "Activate" recalls it.

### MIDI mapping

MIDI mappings are **data, not Entities**. `nap::MidiMapping` (`module/src/midimapping.h/.cpp`) is a declarative resource: a message filter mirroring `nap::MidiInputComponent`'s (`Ports`/`Channels`/`Numbers` + per-message-type booleans) plus a target — `TargetKind` is `Parameter` (drives a `ParameterFloat` with `InputMinimum`/`InputMaximum` normalization), `Preset` (calls `lxcontrolService::applyPreset()`), or `EffectTrigger` (`EffectLayer` + `EMidiTriggerAction` Play/Stop/Toggle, the same enum `MidiTriggerComponent` uses). This replaces the earlier pattern of one hand-authored `Entity` (`MidiInputComponent` + `MidiParameterComponent`/`MidiTriggerComponent`) per mapping — that doesn't support mappings created live, since new Entities can't be cheaply added to a running `Scene`'s static entity list.

Instead, `objects.json` declares exactly **one** always-on, wildcard-filtered `MidiInputComponent` (`MidiMonitorEntity`/`MidiIn_Monitor` — every filter field empty/true, so it receives everything), and `lxcontrolService::setup()` connects to its `messageReceived` signal. `lxcontrolService::onMidiEvent()` logs every message (for the MIDI tab's live log and the Learn workflow, see below) and checks it against every live `MidiMapping::matches()`, applying whichever match it finds. Creating a mapping is a **Learn** workflow in the app's **MIDI** tab: pick a target kind + target, click Learn, and the next incoming message's filter (message type + number, exactly — not port/channel, to keep learned mappings device-agnostic by default) becomes a new `MidiMapping` via `lxcontrolService::createMidiMapping()`. The tab's device list comes from `nap::MidiInputPort::getPortNames()` (the ports the static `MidiPort` resource opened at startup — `napmidi` has no live/hot-plug enumeration, so this is a startup snapshot, not a push notification).

### App / frame lifecycle

`lxcontrolApp` (`src/lxcontrolapp.h/.cpp`) resolves `ParameterGUI`s, `Fixture`s, `ParameterGroup`s, `Scene`/entities, and the wildcard `MidiInputComponentInstance` by `mID`/`findEntity` in `init()`, then calls `lxcontrolService::setup()` once to hand off the fixture rig and MIDI listener. In `update()`, after routing input, it calls `mGuiService->selectWindow(mRenderWindow)` and then `drawMainUI()`, which draws one `ImGui::BeginTabBar` with four tabs (Fixtures / Presets / Effects / MIDI) — this project's own tabbed layout, not the `CollapsingHeader`-accordion style used by NAP's demos, since real tabs fit "one page per area" better. Any effect timelines the user toggled open are kept visible across tab switches by calling their `SequenceEditorGUI::show()` every frame from `update()`, after the tab bar. Fixture/Effect/MIDI-dispatch logic beyond the GUI itself lives in `lxcontrolService` (module layer) and in `FixtureDmxWriterComponent`/`ArtNetController` (per-frame DMX output) — the app itself stays a thin GUI + input + render shell.

`lxcontrolService` (`module/src/lxcontrolservice.h/.cpp`) is **not** a stub — see Presets/Effects/MIDI above. It owns the live lists of Presets/EffectLayers/MidiMappings, all runtime construction and persistence, and MIDI dispatch. Its own `update()` is currently empty; the `show()` calls for open timelines are driven from the app (see above), not the service, since `SequenceEditorGUI::show()` must happen after `selectWindow()` in `App::update()` — see the `IMGuiService::selectWindow` gotcha.

**Required NAP modules**: app (`app.json`) — `napapp`, `napcameracontrol`, `napartnet`, `napmidi`, `napparametergui`, `napsequencegui`, `naplxcontrol`; module (`module/module.json`) — `naprender`, `napscene`, `napparameter`, `napartnet`, `napmidi`, `napsequence`, **`napsequencegui`, `napimgui`** (the latter two added because `lxcontrolService` itself now constructs `SequenceEditorGUI` resources at runtime, not just the app).

**Data-driven vs. runtime-authored objects**: the fixed rig (fixtures, universes, the wildcard MIDI listener, per-fixture GUIs) is edited by hand in `data/objects.json` (or in Napkin) — see `Fixture_Strobe1`/`FixtureType_EuroliteStrobe540_22ch` as the template for a new *kind* of fixture. Everything else — Presets, Effects, MIDI mappings — is created, edited, and deleted **live in the app's own GUI**, never by hand-editing JSON; it persists to `data/user_content.json`, which `lxcontrolService` owns exclusively. Adding a new physical fixture is still a manual `objects.json` edit (duplicate a `Fixture_StrobeN` block, add its writer component and `ParameterGUI`, wire it into `lxcontrolApp::init()`'s fixture/group lookups); a new *kind* of channel type, mixing policy, mapping target-kind, or preset behavior requires a new module class.

**Known simplifications, worth revisiting:**
- "Delete" on a Preset/EffectLayer/MidiMapping only removes it from `lxcontrolService`'s tracked lists (and stops it, for effects) — the underlying resource isn't actually freed from the `ResourceManager`, since `addObject`/`removeObject` are private API only used internally by the JSON-load/diff path. Harmless (it just sits unreferenced in memory until restart), but not a true delete.
- Preset recall is an instant value jump per fixture, not a timed crossfade. The earlier `ParameterBlendComponent`-based approach did support blending but couldn't support a growing, live-created preset list; nothing currently replaces that blend behavior.

## NAP skills & agents

`.claude/skills/` holds NAP Framework skills (foundation: `nap-skill-authoring`, `nap-resource-graph`, `nap-adding-a-module-class`, `nap-build-run-verify`, `nap-validate-data-json`; domains: `nap-rendering`, `nap-sequence`, `nap-parameters`, `nap-artnet-dmx`, `nap-midi`). They keep NAP work grounded in verified sources instead of hallucinated API; the `nap-skill-authoring/scripts/` helpers (`nap_doc`, `nap_usage`, `nap_validate`, `nap_new_component`, `nap_doctor`) do the lookups/scaffolding/linting.

`.claude/agents/` holds NAP agents: **nap-explore** (read-only tracing), **nap-architect** (read-only design), **nap-implement** (writes code/JSON, lints/builds).

**Passing skills to an agent:** each NAP agent loads a default set, and additionally loads any skill named in a `Skills:` line in its task prompt. When dispatching one, add that line so it pulls the domain knowledge the task needs — e.g.:

```
Skills: nap-resource-graph, nap-artnet-dmx
Add a strobe fixture entity to data/objects.json wired to Universe0 …
```

The agent invokes each named skill (or reads `.claude/skills/<name>/SKILL.md`) before starting.
